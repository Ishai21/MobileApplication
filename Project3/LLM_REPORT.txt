Short LLM Usage Report — Project 3 (Chicago Guide)

For this project I used Cursor’s built-in AI assistant (a large language model–based coding assistant) during design, implementation, and debugging. If course policy requires naming a specific underlying model provider, that name can be substituted here without changing the substance of this report.

I used the assistant first to interpret the written assignment (Project 3 description): two conceptual roles (A1 launcher UI vs. A2 detail UI), implicit intents between separate apps, two activities in the receiver app, two fragments per activity (list + WebView), master/detail width behavior, overflow options menu, back-stack behavior, and retaining state across configuration changes. The assistant helped map those requirements onto concrete Android components (activities, fragments, MaterialToolbar, ListView with single-choice highlighting, WebView with saved state, OnBackPressedCallback).

Because Android Studio did not show a second Gradle module cleanly in my local setup, I asked the assistant to consolidate the work into a single application module (`:app`) so one install/run configuration would open the full flow. In that consolidated design, the “A1” screen is `MainActivity` (two descriptive `TextView`s, two buttons, short `Toast`s, then navigation). Navigation to attractions vs. restaurants uses explicit `Intent`s to `AttractionsActivity` and `RestaurantsActivity` in the same package, not separate APKs and not implicit resolution across apps. That is a deliberate deviation from the original “two apps + implicit intents” wording; if the course requires two APKs and implicit intents for grading, the project would need to be split back into two application modules with matching `<intent-filter>` actions and `<queries>` on the sender.

The assistant generated and iterated on Java sources (`MainActivity`, `BaseBrowseActivity`, `AttractionsActivity`, `RestaurantsActivity`, `ItemListFragment`, `WebDetailFragment`), XML layouts (including a `layout-land` variant for the browse screen), the options menu, manifest entries (`INTERNET` for WebView), and Gradle configuration. It also helped troubleshoot a resource-compilation failure (`aaptcompiler` / merged `values.xml` / `string-array` parsing) by moving list data out of `string-array` resources into Java `static final String[]` constants to avoid the buggy merge path, and by moving the assignment text file out of `res/` into `assets/`.

Overall, the LLM outputs were useful for structure and speed but not fully reliable without manual verification. I re-checked behavior against the spec (list selection persistence, 1/3 vs 2/3 split after selection, back returning to list-only, toolbar + overflow menu, WebView state across rotation) and confirmed the implementation in Android Studio. I also verified build settings: Java-only app code, `compileSdk` / `targetSdk` 36, `minSdk` 36 per the assignment’s Android 16 target. Final integration, running on an emulator or device, and any demo video remain my responsibility.
