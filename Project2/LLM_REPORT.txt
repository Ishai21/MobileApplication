Short LLM Usage Report

For this project I used two large language models (LLMs): Cursor’s built-in AI assistant (based on OpenAI models) and a second general-purpose LLM accessed through a web interface (for example, ChatGPT or Gemini). I relied on them at several points in the development process, both for refining the specification and for generating and explaining code. If the exact LLM product names differ, they can be substituted without changing the substance of this report.

I used the LLMs first for spec refinement. Starting from the rough assignment description, I asked for help clarifying what “grid formatting” meant in concrete terms for a Pixel 9 device (number of columns, spacing, cell shape, and scaling behavior). The assistant helped me converge on a 2-column grid, approximately 6 dp spacing, and poster-like cells that preserve aspect ratio using centerCrop for thumbnails and fitCenter for the full-screen poster. It also guided me in writing a refined spec document that explicitly lists the nine movies, their order, and how each URL (IMDb, Rotten Tomatoes, streaming) is mapped to grid positions.

I then used the LLMs for code generation and design. They helped me design the overall architecture (Java-based app with a GridView, BaseAdapter, and a separate PosterActivity), and generated initial versions of MainActivity, MovieGridAdapter, the XML layout files (activity_main.xml, grid_item_movie.xml, activity_poster.xml), and the logic for handling short clicks, context menus, and Intent extras. I also used the assistant to adjust small details (for example, where to register the context menu, how to open URLs safely, and how to pass the IMDb URL to the poster screen) and to check that my refined spec and implementation were consistent.

Overall, the accuracy of the LLM outputs was high but not perfect. The generated code generally compiled and followed Android best practices, but I still had to verify that resource names matched my actual drawable files and that the layouts looked correct on the Pixel 9 AVD. In a few cases, I needed to adjust details (for example, making sure the project used Java rather than Kotlin and aligning the activity and manifest configuration with the template project). The LLMs were especially accurate at boilerplate code and high-level structure, while I provided the final debugging, resource wiring, and visual fine-tuning. Using LLMs in this way significantly sped up development, but I treated their output as a strong starting point that still required careful review and testing.
